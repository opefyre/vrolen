# Audit: UI → engine data flow

Snapshot of every user-facing simulation knob and whether the value the user
typed ends up changing the engine's behavior. Audit run on 2026-06-22 against
the codebase at /Users/abosh/Desktop/Personal/Projects/sc-simulator/vrolen.

Pipeline under audit:

```
EditorPage Inspector / Run-settings drawer
   └─> node.data.* / settings.*
        └─> graphToChainOptions (src/lib/graph-to-chain.ts)
             └─> ChainOptions
                  └─> runChain → simulationStream (src/engine/chain-harness.ts)
                       └─> CycleExecutor + StateMachine + KPI accumulators
```

Two entry points consume the pipeline:

- `EditorPage.handleRun` (src/routes/EditorPage.tsx:1602) — canvas Run button.
- `runScenario` (src/lib/run-scenario.ts) — comparison / saved-scenario replays.

They are not identical. Mismatches between them surface below.

## Summary

| Severity                                   | Count |
| ------------------------------------------ | ----- |
| P0 (math wrong / user mis-led)             | 0     |
| P1 (silently dropped or invisibly skipped) | 6     |
| P2 (UX / cosmetic / dead code)             | 6     |

Top 5 most critical findings:

1. **Linear-fallback in `graph-to-chain.ts` (line 362+) drops every per-station feature.** When the graph isn't a single-source / single-sink DAG (e.g. fan-out, fan-in, multi-source, disconnected branches), the function returns `topology: null` and the engine receives only `stationCycleTimes` + `stationLabels`. Capacity, defectRate, setup, changeoverMatrix, cycleByProduct, reworkTargetId, reworkPassLimit, and skills are all dropped. The recent VROL-627 fix only patched the single-source/single-sink branch — anything outside that envelope still silently mis-runs.
2. **`settings.schedule` (line-wide breaks + maintenance windows) is read by nothing in the run pipeline.** A whole accordion in the Run-settings drawer writes to a field that no run path consumes. The user enables "Line-wide schedule", adds breaks, runs, and gets unchanged numbers.
3. **Fan-out graphs collapse to a one-station chain.** `A → {B, C}` produces `chainNodeIds = ["A"]` because the linear-fallback bails out the moment a node has out-degree != 1. Both B and C — including their cycle times — are silently dropped.
4. **`runScenario` drops `samplerIntervalMs`.** EditorPage's run path forwards it; the comparison path doesn't. Comparing two saved scenarios produces zero timeseries samples — throughput-over-time + WIP-curve cards silently render flat.
5. **Inspector exposes no UI for `changeoverMatrix`.** The engine consumes it (verified end-to-end in presets), but no field in the Inspector lets a user create or edit one. Per-product changeovers can only be authored via a preset or programmatic JSON import.

Status legend:

- **DROPPED** — value is read by some layer but the engine never sees it.
- **MISMATCH** — value reaches the engine but UI promise diverges from engine behavior.
- **UNUSED** — engine receives the value but does nothing with it.
- **OK** — verified end-to-end.
- **NEEDS_SPEC** — engine behavior is well-defined but UX implies functionality that doesn't exist (no per-station MTBF).

## What's verified OK

End-to-end confirmed during the audit:

- Per-station `maintenanceWindows` pause the station for the window duration (audit-8). Engine emits per-station maintenance ms.
- `products.productionPlan` exhausts the source after the listed quantity (audit-9). Engine emits `perProductCompleted`.
- `source.batchSize` pushes N parts per arrival event (audit-10).
- `workers[].skills` mismatch blocks a station (audit-11). Engine surfaces "Starved on no-skill-available".
- `breakdowns.enabled` reaches the engine; `perStationBreakdowns` populated (audit-7 implicit).
- `defectRate` applied to ANY station type (not just QC), driving scrap (cycle-execution.ts:347).
- Replication count 1..50 honored — EditorPage.tsx:1725 clamps and loops `runChain` (no repro written; verified by code reading).
- One-shot `materials.replenishment.atMs` + `amount` fires (settings-to-materials-cfg.ts:50; engine handles via material-replenishment event).
- Recurring `materials.recurring` deliveries fire at intervalMs cadence with `maxInventory` cap respected (chain-harness.ts:1073-1100, settings-to-materials-cfg.ts:60).
- `interStationBufferCapacity` applied uniformly to every edge buffer (chain-harness.ts:782) — no per-edge buffer UI exists or claims to exist.
- Wizard → editor handoff preserves per-station fields the wizard sets (commit-draft.ts:44-56). `defectRate` reaches each node correctly.
- Cost / revenue fields are post-processed by cost-economics.ts — intentional, not engine-side.
- Cycle distribution (constant / uniform / normal / triangular / exponential) per station: DistributionField → node.data.cycleDistribution → graphToChainOptions → engine.

## Notes on the audit

- Audit took only the EditorPage Run + runScenario compare paths. The /run page (RunPage.tsx) and Sensitivity / WIP / Optimization sweeps may have their own divergent code paths; not audited here.
- Bulk-edit (`BulkInspector`) writes through the same `updateSelectedNodeData` plumbing so anything the inspector drops, bulk-edit also drops.
- Validation (`src/lib/validate-scenario.ts`) is upstream of `graphToChainOptions`. A surface that bumps "your rework target isn't on the chain" warning before the run would catch most of the P1 silent drops at edit time instead of after a confusing run.
- The recent VROL-627 fix correctly emits `topology` for branching DAGs. The remaining gap is the fallback branch — extending the DAG branch to handle multi-source / multi-sink graphs (with synthetic source/sink nodes, or by picking the first source / sink) would close most of the P1 list above.
